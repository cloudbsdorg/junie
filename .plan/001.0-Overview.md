## 1. Executive Summary

Junie is an LLM-agnostic coding agent developed by JetBrains that integrates seamlessly with terminals, IDEs, and CI/CD pipelines. Unlike traditional AI assistants that are tied to a single model provider, Junie supports multiple authentication methods including JetBrains Account, API keys, and Bring Your Own Key (BYOK) configurations with providers such as Anthropic, OpenAI, Google, xAI, OpenRouter, and Copilot.

The agent understands natural language instructions and translates them into executable development tasks including code implementation, bug fixes, PR reviews, and repository analysis.

---

## 2. Product Vision

Junie is designed around three core principles:

1. **LLM-Agnostic**: Users can choose their preferred model provider or use JetBrains' default service
2. **Seamless Integration**: Works identically across CLI, IDE, and CI/CD environments
3. **Privacy-First**: Minimal telemetry with transparent opt-out mechanisms

### 3.1 Core Components

| Component | Description | Language/Technology |
|-----------|-------------|---------------------|
| CLI | Terminal-based user interface | Go (cross-platform binary) |
| IDE Plugin | JetBrains IDE integration | Kotlin/Java |
| VS Code Extension | VS Code integration | TypeScript |
| GitHub Action | CI/CD integration | TypeScript (Node.js) |
| Agent Core | Session, context, tool management | Go |
| Provider Adapters | LLM provider communication | Go |

### 3.2 Deployment Model

```
┌─────────────────────────────────────────────────────────────────────┐
│                        End User Machine                              │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  Junie CLI / IDE Plugin / VS Code Extension                   │  │
│  │                                                               │  │
│  │  - Reads repository files locally                             │  │
│  │  - Executes tools (git, shell) on local machine               │  │
│  │  - Sends prompts to LLM provider (HTTPS)                       │  │
│  │  - Receives streamed responses                                │  │
│  └──────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
                                   │
                    (Direct HTTPS to LLM Provider)
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM Provider (e.g., Anthropic, OpenAI)            │
└─────────────────────────────────────────────────────────────────────┘
```

**Note:** Junie processes files locally and only sends prompts/text to LLM providers. Code never leaves the user's machine except when explicitly requested for specific features like GitHub integration.

## 6. Key Features

### 6.1 Natural Language Interface

Users interact with Junie using natural language:

```bash
junie
> Build a web dashboard that shows open issues, recent commits, and CI status for this repository

junie
> Fix the failing tests in the auth module and add a regression test

junie
> Review the current changes in this branch and highlight potential bugs
```

### 6.2 GitHub Integration

Junie can respond to GitHub events automatically when configured as a GitHub Action:

- Issue comments and creation
- PR reviews and commentary
- CI failure notifications
- Commit status updates

### 6.3 Multi-Provider Support

| Provider | Authentication | Models |
|----------|---------------|--------|
| JetBrains AI Hub | OAuth (JetBrains Account) | Default models |
| Anthropic | API Key | Claude family |
| OpenAI | API Key | GPT-4, GPT-4o |
| Google | API Key | Gemini family |
| xAI | API Key | Grok |
| OpenRouter | API Key | Various |
| GitHub Copilot | OAuth | Copilot models |

## 8. Design Principles

1. **Local Processing**: File operations happen locally; only prompts are sent to LLM providers
2. **Provider Abstraction**: Clean interfaces allow adding new LLM providers without core changes
3. **Stateless Core**: Session state is managed separately for testability
4. **Graceful Degradation**: Works offline for local operations; LLM features require connectivity
5. **Incremental Implementation**: Features are built incrementally with backward compatibility
